# Decision Register

These choices were resolved before drafting final policy pages. Suggested defaults are kept as context for future policy changes, not as separate authority; where the research reports marked community consensus, the suggested default says so.

Resolved decisions are policy for drafted pages. New policy questions should be added here before changing policy pages.

| ID | Decision | Options | Suggested default | Resolution |
| --- | --- | --- | --- | --- |
| D1 | PostgreSQL version posture | Pin to PG18+, support PG16+, or version-agnostic | Assume PG18+ for new databases and new DDL; call out version-gated features (`uuidv7()`, virtual generated columns, temporal constraints, `RETURNING OLD/NEW`, `NOT NULL ... NOT VALID`) inline so older targets can adapt | Resolved: Assume PG18+ for new databases and new DDL. Version-gated features (`uuidv7()`, virtual generated columns, temporal constraints, `RETURNING OLD/NEW`, `NOT NULL ... NOT VALID`) are called out in `Version Notes` with fallbacks so older targets can adapt. |
| D2 | Database posture and logic placement | Database-centric (functions/triggers carry business logic), app-first with database integrity, or app-only (no DB constraints) | App-first: business logic lives in the application; the database owns data integrity via constraints; server-side logic (functions, triggers) is minimal and justified case by case | Resolved: App-first: business workflow logic lives in the application. The database owns data integrity (constraints, FKs, CHECKs) plus audit and derived-data triggers per D42. Functions and procedures stay minimal and justified case by case. |
| D3 | Identifier casing and quoting | Ban quoted identifiers, permit selectively, or allow freely | Consensus: lowercase snake_case identifiers only, never quoted (except legacy interop); avoid reserved words (`user`, `order`); stay under 63 bytes; abbreviate only universally understood terms | Resolved: Lowercase snake_case identifiers only, never quoted (except when interoperating with legacy objects that already require quoting). Avoid reserved words as identifiers; stay under 63 bytes; abbreviate only universally understood terms. |
| D4 | Keyword and built-in function casing | UPPERCASE keywords, lowercase keywords, mixed | UPPERCASE SQL keywords with lowercase identifiers and built-in function names (`SELECT date_trunc(...)`); contested in practice, so confirm before drafting | Resolved: UPPERCASE SQL keywords with lowercase identifiers and lowercase built-in function names (`SELECT date_trunc('day', created_at) FROM users`). |
| D5 | Layout: commas, indentation, line length | Trailing commas vs leading commas; river alignment vs simple indentation; 2 vs 4 spaces | Trailing commas; simple left-aligned indentation (no river alignment); 2-space indent; one clause keyword per line for nontrivial queries; break lines around 100 chars | Resolved: Trailing commas; simple left-aligned 2-space indentation (no river alignment); each clause keyword starts a line in nontrivial queries; wrap around 100 characters. |
| D6 | Comment style and COMMENT ON | `--` vs `/* */`; COMMENT ON mandatory, encouraged, or unused | `--` line comments in SQL files; `COMMENT ON` encouraged for non-obvious tables, columns, and functions but not mandatory; never commit commented-out SQL | Resolved: Document in code: `--` line comments in SQL files (block `/* */` only for multi-line headers); `COMMENT ON` is not used. Never commit commented-out SQL. |
| D7 | Table naming and plurality (flagship) | Plural (`users`), singular (`user`), mixed | Plural snake_case table names; junction tables named for the relationship when a natural name exists, otherwise `a_bs` in alphabetical order; no `tbl_` or Hungarian prefixes | Resolved: Plural snake_case table names (`users`, `order_items`). Junction tables take a natural relationship name when one exists, otherwise the two table names in alphabetical order with the last word pluralized. No `tbl_` or Hungarian prefixes. |
| D8 | Column naming | Bare `id` PK vs `table_id`; timestamp suffix style; boolean prefixes | Bare `id` for PKs; FKs named `<referenced_singular>_id`; `_at` suffix for timestamps (`created_at`); `is_`/`has_` prefixes for booleans; descriptive over generic names | Resolved: Bare `id` for PKs; FKs named `<referenced_singular>_id` (`user_id` references `users`); `_at` suffix for timestamps (`created_at`); `is_`/`has_` prefixes for booleans; descriptive names over generic ones. |
| D9 | Constraint and index naming | Rely on auto-generated names, PG default suffix scheme, custom prefix scheme | Consensus: always name constraints and indexes explicitly; use names matching the PG default suffix pattern (`{table}_pkey`, `{table}_{col}_fkey`, `{table}_{cols}_key`, `{table}_{cols}_idx`, `{table}_{col}_check`) so explicit and auto-generated names agree | Resolved: Always name constraints and indexes explicitly, matching the PG default suffix pattern (`{table}_pkey`, `{table}_{col}_fkey`, `{table}_{cols}_key`, `{table}_{cols}_idx`, `{table}_{col}_check`) so explicit and auto-generated names agree. |
| D10 | Schema layout | Single `public` schema, single dedicated app schema, domain schemas per module | Single application schema per database; `public` acceptable for single-app databases if `CREATE` is revoked from `PUBLIC`; split into domain schemas only when the database genuinely hosts multiple domains | Resolved: Single-app databases keep objects in `public` with `CREATE` revoked from `PUBLIC`. Split into domain schemas only when a database genuinely hosts multiple domains. |
| D11 | search_path policy | Schema-qualify everything, rely on search_path, split by context | Application SQL may rely on the default search_path in single-schema databases; migrations, `SECURITY DEFINER` functions, and anything security-sensitive must schema-qualify references or pin `search_path` explicitly | Resolved: Split by context. Application SQL may rely on the default search_path in single-schema databases. Function bodies, and `SECURITY DEFINER` functions especially, must pin `search_path` (`SET search_path = public, pg_temp`) or schema-qualify references. |
| D12 | Primary key strategy (flagship) | `bigint GENERATED ALWAYS AS IDENTITY`, `uuid DEFAULT uuidv7()`, `uuid` v4, dual bigint-internal/uuid-external | `uuid DEFAULT uuidv7()` for new tables (time-ordered, no B-tree fragmentation, safe to expose); `bigint GENERATED ALWAYS AS IDENTITY` where compactness or extreme insert concurrency matters; never `serial`/`bigserial`; never random v4 PKs | Resolved: Use `id uuid DEFAULT uuidv7() PRIMARY KEY` for new tables. Use `bigint GENERATED ALWAYS AS IDENTITY` only where key compactness or extreme insert concurrency demonstrably matters. Never `serial`/`bigserial`; never random v4 UUIDs as PKs. |
| D13 | Surrogate vs natural keys | Natural keys allowed as PKs, surrogate-only, mixed | Surrogate PKs everywhere; enforce natural keys as `UNIQUE` constraints instead | Resolved: Surrogate PKs everywhere (per D12); natural keys are enforced as `UNIQUE` constraints, never as PKs. |
| D14 | Foreign key posture | FKs required everywhere, FKs default with exceptions, soft references common | FKs required for every reference between durable tables; soft references only for high-volume event/log tables with a documented reason | Resolved: FKs required for every reference between durable tables. Soft references only for high-volume append-only event/log tables, with a documented reason. |
| D15 | ON DELETE policy (flagship) | `CASCADE` default, `RESTRICT`/`NO ACTION` default, per-relationship | `RESTRICT` by default so deletes of referenced rows fail loudly; `CASCADE` only where child rows are true compositions owned by the parent; `SET NULL` rarely, with a documented reason | Resolved: `ON DELETE RESTRICT` by default; deletes of referenced rows fail loudly. `CASCADE` only where child rows are true compositions owned by the parent (for example `order_items`). `SET NULL` rarely, with a documented reason. |
| D16 | String types | `text` everywhere, `varchar(n)` for limits, mixed | Consensus: `text` by default; enforce length limits with named `CHECK` constraints, not `varchar(n)`; `char(n)` banned | Resolved: `text` for all strings; enforce business length limits with named `CHECK` constraints (`char_length(...)`), not `varchar(n)`. `char(n)` banned. |
| D17 | Numeric and money values | `numeric`, integer cents, `money`, `float` | `numeric` for money and exact quantities; `money` and floating-point money banned; integer cents acceptable when the application layer standardizes on it | Resolved: `numeric` for money and exact quantities, with a `currency` column when multi-currency. The `money` type and floating-point money are banned. |
| D18 | Integer widths | `integer` default, `bigint` default, per-column | `bigint` for keys, counters, and anything that grows; `integer`/`smallint` only for values with a known small bound | Resolved: `bigint` by default for counters and anything that can grow; `integer`/`smallint` only for values with a known small bound (ages, positions, percentages). |
| D19 | Timestamp policy | `timestamptz` always, `timestamptz` for events only, mixed | Consensus: `timestamptz` for all timestamps; `date` for calendar dates; `timestamp without time zone` only for future wall-clock times where the UTC offset may change (appointments), with a comment | Resolved: `timestamptz` for all timestamps; `date` for calendar dates. One narrow exception: future wall-clock events where the UTC offset may change (appointments) store local time plus zone name, with a comment. |
| D20 | Ranges and temporal modeling | Start/end column pairs, range types, PG18 temporal constraints | Range types (`tstzrange`, `daterange`) for validity periods; PG18 `WITHOUT OVERLAPS`/`PERIOD` constraints where non-overlap must be enforced, replacing GiST exclusion hacks and trigger checks | Resolved: Range types (`tstzrange`, `daterange`) with half-open `[)` bounds for validity periods, not start/end column pairs. PG18 `WITHOUT OVERLAPS`/`PERIOD` constraints where non-overlap must be enforced, replacing GiST exclusion hacks and trigger checks. |
| D21 | JSONB posture (flagship) | Normalized-first, jsonb-friendly, hybrid | Normalized-first: model core business data relationally; `jsonb` for document-shaped, sparse, fast-evolving, or externally sourced payloads; keep queried/joined attributes as real columns; always `jsonb` over `json` | Resolved: Normalized-first: model core business data relationally with real columns, FKs, and constraints. The hybrid hot/cold pattern (hot attributes as columns, remainder in a `jsonb` column) is acceptable when the data source is external (for example imported data) to manage migration and schema churn. Keep queried, joined, or constrained attributes as real columns. Always `jsonb` over `json`. |
| D22 | Array posture | Arrays freely, arrays for primitive lists only, child tables always | Arrays only for flat lists of primitives with no FK or per-element metadata needs (tags); child tables otherwise; prefer one `jsonb` column over `jsonb[]` | Resolved: Arrays only for flat lists of primitives with no FK targets and no per-element metadata (for example `tags text[]`), GIN-indexed when searched. Child or junction tables otherwise. Prefer one `jsonb` column over `jsonb[]`. |
| D23 | Closed value sets (flagship) | Native enum, lookup table, `text` + `CHECK` | Lookup tables for vocabularies that evolve or carry attributes; `text` plus named `CHECK` for small stable sets; native enums only for truly static universal sets (no `DROP VALUE` support makes them migration-hostile) | Resolved: `text` plus a named `CHECK` constraint is the default for app-owned state machines (for example order status). Use a lookup table with FKs when values carry attributes or the closed set is large (for example US states and territories). Native enums only for truly static universal sets (for example days of the week); their lack of `DROP VALUE` makes them migration-hostile. |
| D24 | Domain types | Use domains liberally, sparingly, never | Domains sparingly, for scalar validation genuinely repeated across tables (email format); prefer plain named `CHECK` constraints for one-off rules | Resolved: Do not use `CREATE DOMAIN`. Use plain types with named per-column `CHECK` constraints, even when a rule repeats across tables. |
| D25 | Lifecycle columns | Required set and maintenance mechanism (trigger vs application) | `created_at`/`updated_at` as `timestamptz NOT NULL DEFAULT now()` on durable business tables; `updated_at` maintained by the application or framework, not by triggers | Resolved: `created_at`/`updated_at` as `timestamptz NOT NULL DEFAULT now()` on durable business tables. `updated_at` maintained by a shared `BEFORE UPDATE` trigger function (per D42), so every write path keeps it correct. |
| D26 | Soft delete vs hard delete (flagship) | Hard delete, `deleted_at` soft delete, archive tables | Hard delete by default; `deleted_at` soft delete only where the product needs recovery or audit, always paired with partial unique indexes (`WHERE deleted_at IS NULL`) and an explicit purge/retention plan | Resolved: Hard delete by default. Adopt `deleted_at` soft delete per table only where the product needs recovery or audit history, always paired with partial unique indexes (`WHERE deleted_at IS NULL`) and an explicit purge/retention plan. |
| D27 | Generated columns | Virtual default, stored default, avoid | Virtual (PG18 default) for cheap derivations; `STORED` when the value is indexed, hot on the read path, or must flow through logical replication; prefer generated columns over update triggers for derivations | Resolved: Virtual (PG18 default) for cheap derivations; explicitly `STORED` when the value is indexed, hot on the read path, expensive to compute, or must flow through logical replication. Prefer generated columns over derivation triggers where the rule is expressible. |
| D28 | NULL policy | `NOT NULL` by default, nullable by default, per-column debate | `NOT NULL` unless absence is semantically meaningful; no sentinel values (empty string, epoch) in place of NULL; `NULLS NOT DISTINCT` used deliberately where NULL should count as a value for uniqueness | Resolved: `NOT NULL` unless absence is semantically meaningful (`confirmed_at`, `deleted_at`). No sentinel values (empty string, epoch) in place of NULL. `NULLS NOT DISTINCT` only as a deliberate choice where NULL should count as a value for uniqueness. |
| D29 | CHECK constraint posture | Encode domain rules in DB, keep validation app-only, minimal checks | Encode cheap, stable domain rules as named `CHECK` constraints (positive amounts, `updated_at >= created_at`); leave complex or fast-changing validation to the application | Resolved: Encode cheap, stable, single-row domain rules as named `CHECK` constraints (positive amounts, length limits, status values, `updated_at >= created_at`). Complex or fast-changing validation stays in the application. |
| D30 | Indexing doctrine | Index speculatively, index FKs + unique keys only, minimal | Consensus: index every FK column; unique business keys get unique constraints; add further indexes only for known query patterns, never speculatively | Resolved: Index every FK column at creation; unique business keys get unique constraints. Add further indexes only for known query patterns, never speculatively. |
| D31 | Advanced index policy | Free use, gated on stated query pattern, avoid | Partial, expression, multicolumn, covering, and GIN indexes allowed only with a stated query pattern; GIN required for searchable `jsonb`/array columns (with containment operators); multicolumn order: most-selective, most-filtered columns first | Resolved: Partial, expression, multicolumn, covering (`INCLUDE`), and GIN indexes allowed only with a stated query pattern (in the migration or commit). GIN is the default for searched `jsonb`/array columns, queried with containment operators (`@>`/`<@`). Multicolumn order: equality-filtered, most-selective columns first. |
| D32 | Unique constraint vs unique index | Constraint always, index always, split by capability | `UNIQUE` constraint by default (visible in `\d`, FK-referenceable); standalone `CREATE UNIQUE INDEX` when partial or expression uniqueness is needed | Resolved: `UNIQUE` constraint by default (visible in `\d`, FK-referenceable, states intent); standalone `CREATE UNIQUE INDEX` only for partial uniqueness (soft delete) or expression uniqueness (`lower(email)`). |
| D33 | Join style | Explicit `JOIN ... ON` only, allow `USING`, allow comma joins | Consensus: explicit ANSI `JOIN ... ON`; comma joins and `NATURAL JOIN` banned; prefer `ON` over `USING`; qualify every column in multi-table queries | Resolved: Explicit ANSI `JOIN ... ON` only; comma joins and `NATURAL JOIN` banned; prefer `ON` over `USING`; qualify every column in multi-table queries. |
| D34 | Alias conventions | Mandatory aliases, aliases when needed, single letters allowed | Short meaningful aliases (first word or initialism of the table); `AS` required for column aliases; no single-letter aliases in queries with more than two tables | Resolved: Short meaningful aliases derived from the table name (`users` -> `u`, `order_items` -> `oi`); single letters acceptable in one- or two-table queries, banned in larger ones; `AS` required for column aliases. |
| D35 | SELECT * policy | Banned in application SQL, allowed, allowed in CTE tails | `SELECT *` banned in application queries, views, and functions; acceptable in ad hoc/exploratory SQL and `EXISTS (SELECT 1 ...)` bodies | Resolved: `SELECT *` banned in application queries, views, and functions; explicit column lists required. Acceptable in ad hoc/exploratory SQL, `EXISTS (SELECT 1 ...)` bodies, and `count(*)`. |
| D36 | Subquery idioms | EXISTS-first, IN-first, mixed | Consensus: `EXISTS (SELECT 1 ...)` for semi-joins; `NOT EXISTS` over `NOT IN` (NULL trap); `= ANY(array)` for parameterized lists; `LATERAL` for per-row top-N and derived computations | Resolved: `EXISTS (SELECT 1 ...)` for semi-joins; `NOT EXISTS` always over `NOT IN` against subqueries (NULL trap); `= ANY(array)` for parameterized lists; `LATERAL` for per-row top-N and derived computations. |
| D37 | CTE decomposition threshold (flagship) | Monolithic queries, CTE-decomposed, threshold-based | Clarity-first: decompose any query with more than one level of nesting or more than one logical step into sequential CTEs; rely on PG12+ inlining; write `MATERIALIZED`/`NOT MATERIALIZED` only as a deliberate, commented choice | Resolved: Clarity-first decomposition. Any query with more than one level of nesting or more than one logical step is refactored into sequential named CTEs, each one comprehensible unit. Rely on PG12+ inlining; write `MATERIALIZED`/`NOT MATERIALIZED` only as a deliberate, commented choice. Simple queries stay plain. |
| D38 | Ordering and pagination | Offset pagination, keyset pagination, per-case | Explicit `ORDER BY` (with a deterministic tiebreaker) whenever `LIMIT` is present; keyset pagination for application-facing lists; `OFFSET` only for small, bounded admin/reporting pages; window functions over self-joins for ranking | Resolved: Explicit `ORDER BY` with a deterministic tiebreaker (append `id`) whenever `LIMIT` is present. Keyset pagination for application-facing lists; `OFFSET` only for small, bounded admin/reporting pages. Window functions over self-joins for ranking. |
| D39 | Upsert strategy | `INSERT ... ON CONFLICT`, `MERGE`, per-case | `INSERT ... ON CONFLICT` for single-row and simple batch upserts; `MERGE` for multi-action bulk synchronization (insert/update/delete in one pass) | Resolved: `INSERT ... ON CONFLICT (cols) DO UPDATE` for row-level and simple batch upserts; `MERGE` for multi-action bulk synchronization where one pass must insert, update, and delete against a source set. |
| D40 | DML safety and RETURNING | Conventions for UPDATE/DELETE and post-write reads | Every `UPDATE`/`DELETE` carries a `WHERE` clause (or an explicit full-table comment); use `RETURNING` instead of follow-up SELECTs; adopt PG18 `RETURNING OLD/NEW` where deltas matter; bulk changes via data-modifying CTEs or batched statements | Resolved: Every `UPDATE`/`DELETE` carries a `WHERE` clause; intentional full-table writes need an explicit marker comment. Use `RETURNING` instead of follow-up SELECTs; adopt PG18 `RETURNING OLD/NEW` where deltas matter. Bulk changes via data-modifying CTEs or batched statements, not row-at-a-time loops. |
| D41 | Function conventions | Language, volatility, and security defaults | Minimal database functions; plain SQL functions over PL/pgSQL when the body is a single query; always label the strictest correct volatility (`IMMUTABLE`/`STABLE`/`VOLATILE`); `SECURITY INVOKER` default, `SECURITY DEFINER` only with pinned `search_path`; procedures only when transaction control is required | Resolved: Minimal database functions (trigger functions per D42/D25 are the main population). Plain SQL functions over PL/pgSQL when the body is a single query. Always label the strictest correct volatility (`IMMUTABLE`/`STABLE`/`VOLATILE`). `SECURITY INVOKER` default; `SECURITY DEFINER` only with pinned `search_path` (D11). Procedures only when transaction control inside the body is genuinely needed. |
| D42 | Trigger policy (flagship) | Trigger-friendly, minimal-trigger, banned | Minimal-trigger doctrine: no business logic in triggers; prefer constraints and generated columns; triggers acceptable only for cross-table auditing or invariants constraints cannot express, and each one needs a comment justifying it | Resolved: Triggers are the sanctioned mechanism for audit tables and derived-data maintenance (for example `updated_at`, denormalized counters), guaranteeing coverage regardless of write path. No business workflow logic in triggers; prefer constraints and generated columns where they can express the rule. Name triggers consistently and keep each trigger body small and single-purpose. |
| D43 | View policy | Views liberally, views sparingly, app-side composition only | Views sparingly, for genuinely stable read models or column masking; every view column explicitly aliased; materialized views only with a documented refresh strategy and the unique index `REFRESH ... CONCURRENTLY` requires | Resolved: Views sparingly, for genuinely stable read models or column masking, with one sanctioned pattern: read-only advanced analytics reports should become views so they can be exposed to different reader surfaces (apps, admin, BI). Every view column explicitly aliased; no `SELECT *`. Materialized views only with a documented refresh strategy and the unique index `REFRESH ... CONCURRENTLY` requires. |
| D44 | Role topology and grants | Single app role, small standard topology, per-service roles | Small standard topology: object-owning migration role, app runtime role (DML only, no DDL), read-only role for reporting/analytics; deny-by-default with `ALTER DEFAULT PRIVILEGES`; grant to roles, never to individual users; app never connects as owner or superuser | Resolved: Three-role topology: object-owning migration role (DDL), app runtime role (DML only, no DDL), read-only role for reporting/analytics. Deny-by-default with `ALTER DEFAULT PRIVILEGES`; grant to group roles, never to individual users; the app never connects as owner or superuser. |
| D45 | Row-level security posture | RLS by default, RLS for multi-tenant only, avoid RLS | No RLS by default; adopt RLS when multi-tenant isolation or row-level access rules are a real requirement, with policies reviewed alongside the role topology | Resolved: Avoid RLS. Tenant and row scoping is enforced in the application query layer; database-side isolation, if ever required, is a deliberate architecture decision outside this guide's defaults. |
| D46 | Safe migration mechanics | Strict zero-downtime rules vs lenient | Consensus: expand/contract for breaking changes; `CREATE INDEX CONCURRENTLY` outside transactions; constraints added `NOT VALID` then `VALIDATE` (PG18: `NOT NULL` too); batched backfills; short `lock_timeout` in every migration; multi-release column-drop protocol | Resolved: Strict zero-downtime rules for production databases: expand/contract for breaking changes; `CREATE INDEX CONCURRENTLY` outside transactions; constraints added `NOT VALID` then `VALIDATE` separately (PG18: `NOT NULL` too); batched backfills; short `lock_timeout` in every migration; multi-release column-drop protocol. |
| D47 | Migration files and tooling | Framework-native migrations, versioned SQL files, tooling-agnostic rules | Follow the project's framework-native migration tool when one exists; otherwise sequential versioned SQL files; decide per project whether down-migrations are required and whether a safe-migration linter (strong_migrations-style) is mandated | Resolved: Follow the project's framework-native migration tool and its file conventions when one exists; otherwise sequential versioned SQL files. Down-migrations and safe-migration linters (strong_migrations-style) are per-project choices. Guide examples stay raw SQL. |
