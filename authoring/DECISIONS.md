# Decision Register

This register contains the current policy decisions for drafted pages. Historical options and rejected defaults live in git history and the source research, not beside active policy.

Add or amend a resolution here before changing policy pages.

| ID | Decision | Resolution |
| --- | --- | --- |
| D1 | PostgreSQL version posture | Assume PG18+ for new databases and new DDL. Version-gated features (`uuidv7()`, virtual generated columns, temporal constraints, `RETURNING OLD/NEW`, `NOT NULL ... NOT VALID`) are called out in `Version Notes` with fallbacks so older targets can adapt. |
| D2 | Database posture and logic placement | App-first: business workflow logic lives in the application. The database owns data integrity (constraints, FKs, CHECKs) plus audit and derived-data triggers per D42. Functions and procedures stay minimal and justified case by case. |
| D3 | Identifier casing and quoting | Lowercase snake_case identifiers only, never quoted (except when interoperating with legacy objects that already require quoting). Avoid reserved words as identifiers; stay under 63 bytes; abbreviate only universally understood terms. |
| D4 | Keyword and built-in function casing | UPPERCASE SQL keywords with lowercase identifiers and lowercase built-in function names (`SELECT date_trunc('day', created_at) FROM users`). |
| D5 | Layout: commas, indentation, line length | Trailing commas; simple left-aligned 2-space indentation (no river alignment); each clause keyword starts a line in nontrivial queries; wrap around 100 characters. |
| D6 | Comment style and COMMENT ON | Document in code: `--` line comments in SQL files (block `/* */` only for multi-line headers); `COMMENT ON` is not used. Never commit commented-out SQL. |
| D7 | Table naming and plurality (flagship) | Plural snake_case table names (`users`, `order_items`). Junction tables take a natural relationship name when one exists, otherwise the two table names in alphabetical order with the last word pluralized. No `tbl_` or Hungarian prefixes. |
| D8 | Column naming | Bare `id` for PKs; FKs named `<referenced_singular>_id` (`user_id` references `users`); `_at` suffix for timestamps (`created_at`); `is_`/`has_` prefixes for booleans; descriptive names over generic ones. |
| D9 | Constraint and index naming | Use the PostgreSQL default suffix pattern (`{table}_pkey`, `{table}_{col}_fkey`, `{table}_{cols}_key`, `{table}_{cols}_idx`, `{table}_{col}_check`). Omit an explicit constraint name only when PostgreSQL will generate that canonical name unambiguously; otherwise name it explicitly. Index names are always explicit. In ORM-managed schemas, follow the surrounding ORM convention. |
| D10 | Schema layout | Single-app databases keep objects in `public` with `CREATE` revoked from `PUBLIC`. Split into domain schemas only when a database genuinely hosts multiple domains. |
| D11 | search_path policy | Split by context. Application SQL may rely on the default search_path in single-schema databases. Function bodies must pin `search_path` (`SET search_path = public, pg_temp`); `SECURITY DEFINER` functions must also schema-qualify object references. |
| D12 | Primary key strategy (flagship) | Use `id uuid DEFAULT uuidv7() PRIMARY KEY` for new tables. Use `bigint GENERATED ALWAYS AS IDENTITY` only where key compactness or extreme insert concurrency demonstrably matters. Never `serial`/`bigserial`; never random v4 UUIDs as PKs. |
| D13 | Surrogate vs natural keys | Surrogate PKs everywhere (per D12); natural keys are enforced as `UNIQUE` constraints, never as PKs. |
| D14 | Foreign key posture | FKs required for every reference between durable tables. Soft references only for high-volume append-only event/log tables and CHECK-constrained polymorphic type-and-ID references (actor/subject columns), with a documented reason. |
| D15 | ON DELETE policy (flagship) | `ON DELETE RESTRICT` by default; deletes of referenced rows fail loudly. `CASCADE` only where child rows are true compositions owned by the parent (for example `order_items`). `SET NULL` rarely, with a documented reason. |
| D16 | String types | `text` for all strings; enforce business length limits with named `CHECK` constraints (`char_length(...)`), not `varchar(n)`. `char(n)` banned. |
| D17 | Numeric and money values | `numeric` for money and exact quantities, with a `currency` column when multi-currency. The `money` type and floating-point money are banned. |
| D18 | Integer widths | `bigint` by default for counters and anything that can grow; `integer`/`smallint` only for values with a known small bound (ages, positions, percentages). |
| D19 | Timestamp policy | `timestamptz` for all timestamps; `date` for calendar dates. One narrow exception: future wall-clock events where the UTC offset may change (appointments) store local time plus zone name, with a comment. |
| D20 | Ranges and temporal modeling | Range types (`tstzrange`, `daterange`) with half-open `[)` bounds for validity periods, not start/end column pairs. PG18 `WITHOUT OVERLAPS`/`PERIOD` constraints where non-overlap must be enforced, replacing GiST exclusion hacks and trigger checks. |
| D21 | JSONB posture (flagship) | Normalized-first: model core business data relationally with real columns, FKs, and constraints. The hybrid hot/cold pattern (hot attributes as columns, remainder in a `jsonb` column) is acceptable when the data source is external (for example imported data) to manage migration and schema churn. Keep queried, joined, or constrained attributes as real columns. Always `jsonb` over `json`. |
| D22 | Array posture | Arrays only for flat lists of primitives with no FK targets and no per-element metadata (for example `tags text[]`), GIN-indexed when searched. One carve-out: write-once ID arrays may reference tables that are never hard-deleted, with a documented reason; a junction table is still required when membership carries authority, per-element attributes beyond order, or row-level constraints. Child or junction tables otherwise. Prefer one `jsonb` column over `jsonb[]`. |
| D23 | Closed value sets (flagship) | `text` plus a named `CHECK` constraint is the default for app-owned state machines (for example order status). Use a lookup table with FKs when values carry attributes or the closed set is large (for example US states and territories). Native enums only for truly static universal sets (for example days of the week); their lack of `DROP VALUE` makes them migration-hostile. |
| D24 | Domain types | Do not use `CREATE DOMAIN`. Use plain types with named per-column `CHECK` constraints, even when a rule repeats across tables. |
| D25 | Lifecycle columns | `created_at`/`updated_at` as `timestamptz NOT NULL DEFAULT now()` on durable business tables. `updated_at` maintained by a shared `BEFORE UPDATE` trigger function (per D42), so every write path keeps it correct. |
| D26 | Soft delete vs hard delete (flagship) | Hard delete by default. Adopt `deleted_at` soft delete per table only where the product needs recovery or audit history, always paired with partial unique indexes (`WHERE deleted_at IS NULL`) and an explicit purge/retention plan. |
| D27 | Generated columns | Virtual (PG18 default) for cheap derivations; explicitly `STORED` when the value is indexed, hot on the read path, expensive to compute, or must flow through logical replication. Prefer generated columns over derivation triggers where the rule is expressible. |
| D28 | NULL policy | `NOT NULL` unless absence is semantically meaningful (`confirmed_at`, `deleted_at`). No sentinel values (empty string, epoch) in place of NULL. `NULLS NOT DISTINCT` only as a deliberate choice where NULL should count as a value for uniqueness. |
| D29 | CHECK constraint posture | Encode cheap, stable, single-row domain rules as named `CHECK` constraints (positive amounts, length limits, status values, `updated_at >= created_at`). Complex or fast-changing validation stays in the application. |
| D30 | Indexing doctrine | Index every FK column at creation; unique business keys get unique constraints. Add further indexes only for known query patterns, never speculatively. |
| D31 | Advanced index policy | Partial, expression, multicolumn, covering (`INCLUDE`), and GIN indexes allowed only with a stated query pattern (in the migration or commit). GIN is the default for searched `jsonb`/array columns, queried with containment operators (`@>`/`<@`). Multicolumn order: equality-filtered, most-selective columns first. |
| D32 | Unique constraint vs unique index | `UNIQUE` constraint by default (visible in `\d`, FK-referenceable, states intent); standalone `CREATE UNIQUE INDEX` only for partial uniqueness (soft delete) or expression uniqueness (`lower(email)`). |
| D33 | Join style | Explicit ANSI `JOIN ... ON` only; comma joins and `NATURAL JOIN` banned; prefer `ON` over `USING`; qualify every column in multi-table queries. |
| D34 | Alias conventions | Short meaningful aliases derived from the table name (`users` -> `u`, `order_items` -> `oi`); single letters acceptable in one- or two-table queries, banned in larger ones; `AS` required for column aliases. |
| D35 | SELECT * policy | `SELECT *` is banned in application queries, views, and functions; explicit column lists are required. It is acceptable only in ad hoc/exploratory SQL. `count(*)` remains the row-count idiom, and committed `EXISTS` predicates use `SELECT 1` per D36. |
| D36 | Subquery idioms | `EXISTS (SELECT 1 ...)` for semi-joins; `NOT EXISTS` always over `NOT IN` against subqueries (NULL trap); `= ANY(array)` for parameterized lists; `LATERAL` for per-row top-N and derived computations. |
| D37 | CTE decomposition threshold (flagship) | Clarity-first decomposition. Any query with more than one level of nesting or more than one logical step is refactored into sequential named CTEs, each one comprehensible unit. Rely on PG12+ inlining; write `MATERIALIZED`/`NOT MATERIALIZED` only as a deliberate, commented choice. Simple queries stay plain. |
| D38 | Ordering and pagination | Explicit `ORDER BY` with a deterministic tiebreaker (append `id`) whenever `LIMIT` is present. Keyset pagination for application-facing lists; `OFFSET` only for small, bounded admin/reporting pages. Window functions over self-joins for ranking. |
| D39 | Upsert strategy | `INSERT ... ON CONFLICT (cols) DO UPDATE` for row-level and simple batch upserts; `MERGE` for multi-action bulk synchronization where one pass must insert, update, and delete against a source set. |
| D40 | DML safety and RETURNING | Every `UPDATE`/`DELETE` carries a `WHERE` clause; intentional full-table writes need an explicit marker comment. Use `RETURNING` instead of follow-up SELECTs; adopt PG18 `RETURNING OLD/NEW` where deltas matter. Bulk changes via data-modifying CTEs or batched statements, not row-at-a-time loops. |
| D41 | Function conventions | Minimal database functions (trigger functions per D42/D25 are the main population). Plain SQL functions over PL/pgSQL when the body is a single query. Always label the strictest correct volatility (`IMMUTABLE`/`STABLE`/`VOLATILE`). `SECURITY INVOKER` default; `SECURITY DEFINER` only with pinned `search_path` (D11). Procedures only when transaction control inside the body is genuinely needed. |
| D42 | Trigger policy (flagship) | Triggers are the sanctioned mechanism for audit tables and derived-data maintenance (for example `updated_at`, denormalized counters), guaranteeing coverage regardless of write path. No business workflow logic in triggers; prefer constraints and generated columns where they can express the rule. Name triggers consistently and keep each trigger body small and single-purpose. |
| D43 | View policy | Views sparingly, for genuinely stable read models or column masking, with one sanctioned pattern: read-only analytics reports consumed by more than one surface (apps, admin, BI) should become views. Every view column explicitly aliased; no `SELECT *`. Materialized views only with a documented refresh strategy and the unique index `REFRESH ... CONCURRENTLY` requires. |
| D44 | Role topology and grants | Three-role topology: object-owning migration role (DDL; a login role whose only client is the migration pipeline, owning the database so it holds `CREATE` on `public` under PG15+), app runtime role (DML only, no DDL), read-only role for reporting/analytics. Deny-by-default with `ALTER DEFAULT PRIVILEGES`, including `REVOKE EXECUTE ON FUNCTIONS FROM PUBLIC`; sensitive tables carved out of the read-only grant and audit tables out of the app role's write grant per table; grant to group roles, never to individual users; the app never connects as owner or superuser. |
| D45 | Row-level security posture | Avoid RLS. Tenant and row scoping is enforced in the application query layer; database-side isolation, if ever required, is a deliberate architecture decision outside this guide's defaults. |
| D46 | Safe migration mechanics | Strict zero-downtime rules for production databases: expand/contract for breaking changes; `CREATE INDEX CONCURRENTLY` outside transactions; constraints added `NOT VALID` then `VALIDATE` separately (PG18: `NOT NULL` too); batched backfills; short `lock_timeout` in every migration; multi-release column-drop protocol. |
| D47 | Migration files and tooling | Follow the project's framework-native migration tool and its file conventions when one exists; otherwise sequential versioned SQL files. Down-migrations and safe-migration linters (strong_migrations-style) are per-project choices. Guide examples stay raw SQL. |
| D48 | Standing session timeouts | Standing `statement_timeout` and `idle_in_transaction_session_timeout` on every runtime login role via `ALTER ROLE ... SET` (settings bind to the login role, not group membership). Never on `{app}_owner` or database-wide: migrations are guarded by `lock_timeout` (D46), and backfills and `VALIDATE CONSTRAINT` legitimately run long. Known long jobs override per session. Add `transaction_timeout` alongside on PG17+ targets. |
| D49 | Public identifiers | Expose the uuid `id` directly in APIs and URLs by default. When the product requires prefixed identifiers (`usr_V1StGXR8Z5`), add `public_id text NOT NULL` with a unique constraint: the type prefix plus an independently generated random suffix. The uuid PK stays internal for all FKs; APIs and URLs then expose only `public_id`. Never derive `public_id` from the PK bits (TypeID-style) or vice versa; the two are joined by an indexed lookup. |
